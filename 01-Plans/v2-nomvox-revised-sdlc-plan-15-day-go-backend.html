Step 1 — Keyword Intake Form (the critical first step)

Instead of a free-text box, the user fills a structured form with 8 guided keyword fields. Each field has inline hint text showing exactly what to type. All fields except "Core idea" are optional but encouraged — the form shows a real-time "prompt richness" indicator that goes from grey to green as more fields are filled, motivating completeness without being mandatory.

On submit, the frontend assembles a single deterministic prompt string and sends it to POST /api/generate. The Go server uses this structured payload — no ambiguous free-text parsing needed. This means the LLM gets enough context on the first call, eliminating the "thin prompt → bad names → user abandons" loop that wastes tokens and session memory.

Prompt richness meter: 1 field = 20% (red), 3 fields = 50% (orange), 5+ fields = 80%+ (green). A tooltip says "More context = better names on the first try, fewer wasted regenerations." The Go handler validates that at minimum the core_idea field is non-empty before calling the LLM.

2 · Availability Scoring & 80% Gate

Every generated name is immediately run through the Go availability engine before it is sent to the frontend. Names that do not score ≥ 80% available are silently discarded — the user only sees names that pass. The score badge (e.g., "100%" or "83%") is shown on each result card. A short explanation appears next to the score (e.g., "Domain + 4/5 handles available").

Scoring formula

PlatformWeightRationale
.com domain2 pointsHighest priority — most founders need a .com
Instagram handle1 pointTier-1 social platform
X (Twitter) handle1 pointTier-1 social platform
TikTok handle1 pointTier-1 for most consumer brands
Threads handle0.5 pointTier-2, growing but not critical
YouTube handle0.5 pointTier-2

Total possible = 6 points. Score % = (points earned / 6) × 100. Threshold: ≥ 80% = pass (≥ 4.8 points). If the .com is unavailable, max score is 4/6 = 67% → automatic fail regardless of social handles. This ensures no name slips through without domain availability.

5 · Key Go Handler Logic

availability/scorer.go — 80% gate

type PlatformResult struct {
    Domain   bool // .com
    Instagram bool
    X         bool
    TikTok    bool
    Threads   bool
    YouTube   bool
}

// Score returns a 0–100 float and whether it passes the 80% threshold.
func Score(r PlatformResult) (float64, bool) {
    points := 0.0
    if r.Domain    { points += 2.0 }  // weighted double
    if r.Instagram { points += 1.0 }
    if r.X         { points += 1.0 }
    if r.TikTok    { points += 1.0 }
    if r.Threads   { points += 0.5 }
    if r.YouTube   { points += 0.5 }
    score := (points / 6.0) * 100.0
    return score, score >= 80.0
}

availability/checker.go — parallel probes

func CheckAll(ctx context.Context, name string) PlatformResult {
    var wg sync.WaitGroup
    var mu sync.Mutex
    r := PlatformResult{}

    probes := []struct {
        key  string
        fn   func() bool
    }{
        {"domain",    func() bool { return checkDomain(ctx, name) }},
        {"instagram", func() bool { return checkSocial(ctx, "https://instagram.com/"+name) }},
        {"x",         func() bool { return checkSocial(ctx, "https://x.com/"+name) }},
        {"tiktok",    func() bool { return checkSocial(ctx, "https://tiktok.com/@"+name) }},
        {"threads",   func() bool { return checkSocial(ctx, "https://threads.net/@"+name) }},
        {"youtube",   func() bool { return checkSocial(ctx, "https://youtube.com/@"+name) }},
    }

    for _, p := range probes {
        wg.Add(1)
        go func(probe struct { key string; fn func() bool }) {
            defer wg.Done()
            result := probe.fn()
            mu.Lock()
            defer mu.Unlock()
            switch probe.key {
            case "domain":    r.Domain = result
            case "instagram": r.Instagram = result
            case "x":         r.X = result
            case "tiktok":    r.TikTok = result
            case "threads":   r.Threads = result
            case "youtube":   r.YouTube = result
            }
        }(p)
    }
    wg.Wait()
    return r
}

9 · Gap Analysis & Solutions (Updated)

GAP 1 — Zero all-green results
Solution: Go server auto-retries Granite (up to 3 cycles) requesting more invented/coined words. After 3 failures, shows highest-scoring name with a breakdown card ("Missing only X handle — try @FolaroHQ") and a "try a different direction" prompt.
GAP 2 — Regenerate memory
Solution: Redis session injects liked[], rejected[], direction_notes[] into every Granite call system prompt. allGenerated[] explicitly excluded. Direction slider value appended as a tonal instruction.
GAP 3 — Thin first-pass prompts
Solution (new): Structured 8-field intake form with hints. Prompt richness meter motivates completeness. Go handler assembles a deterministic, rich prompt on first submit — no multi-round clarification needed.
GAP 4 — Handle probe accuracy
Solution: Go parallel probes with 5s per-probe timeout. 429/5xx = unknown → "check manually" badge. RDAP for domains (not scraping). Redis cache 10 min. Confidence label shown on unknown results.
GAP 5 — Mockup scope creep
Solution: Hard-scoped to single hero-section. Granite generates <section> HTML/CSS only. Rendered in sandboxed <iframe>. No multi-page build.
GAP 6 — Logo quality
Solution: 3 explicit variants with negative prompts for clarity. Per-variant regenerate without regenerating the others. User picks one to include in export.
GAP 7 — Screenshot of taken handles
Solution: Puppeteer server-side crop on taken handles → signed Vercel Blob URL. 10s timeout; silent fallback to "view live" link. Implemented as Day 14 stretch (after core MVP is done).
GAP 8 — 80% threshold edge cases
Solution: User can toggle which platforms matter (e.g., "I don't need YouTube"). Toggle updates the denominator of the score formula in real time. Domain remains double-weighted and cannot be toggled off.
