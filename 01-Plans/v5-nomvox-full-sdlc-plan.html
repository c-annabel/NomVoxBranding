03 Image Generation Decision — Imagen → CSS/SVG

⚠ Why Imagen 3 Was Replaced
The original architecture relied on Google AI Studio free-tier image generation. Three compounding problems made it non-viable for the production demo.
  • Google AI Studio credits exhausted permanently — Free-tier prepaid credits for Imagen/Gemini image generation depleted. Re-enabling requires billing on Google Cloud. Not an option for a hackathon prototype.
  • IBM has no native image generation model on watsonx.ai — IBM Granite is excellent for text. There is no IBM-equivalent of DALL-E, Imagen, or Stable Diffusion accessible through the same watsonx interface.
  • Fallback SVG from LLM hits 429 too — watsonx free tier = 2 req/s. NomVox sends 7 concurrent visual calls simultaneously. All fail with 429 RESOURCE_EXHAUSTED. Made sequential with delays = 20+ second generation wait.
✦ What Was Built Instead — CSS/SVG Visual System
extractPalette() parses the user's color_mood field (bigram matching: "bright orange", "sky blue", etc.) and derives bg, accent, accent2, text tokens. Every visual component renders exclusively from these tokens.

Session Flow
  • UUID v4 session on first generate call
  • Liked/rejected/notes stored in Redis
  • All generation calls inject full session state
  • Names in AllGenerated[] never repeated
  • 2-hour TTL; graceful expiry UI
Availability Scoring
  • Domain (RDAP): 2 points
  • Instagram: 1 point
  • X (Twitter): 1 point
  • TikTok: 1 point
  • Threads: 0.5 pt · YouTube: 0.5 pt
  • Total: 6 pts — threshold: 3.6 (60%)
Visual Pipeline
  • Phase 1: logos + moodboard tiles (concurrent)
  • Phase 2: persona + mockup (sequential)
  • watsonx delay: 600ms between calls
  • CSS/SVG fallback: always shown (primary)
  • AI SVG: overlaid if valid data URI >200 chars
Export ZIP
  • archive/zip streamed (no temp files)
  • SVG-aware: inline SVG written as .svg
  • Includes: brand-brief.json, .html, logos/, mood-board/, README.txt
  • Content-Disposition: attachment

06 Fly.io + Vercel Deployment

Fly.io — Go API
  • Multi-stage Dockerfile: golang:1.22-alpine build + alpine:3.20 runtime
  • fly launch --name nomvox-api --region iad
  • Secrets: WATSONX_API_KEY, WATSONX_PROJECT_ID, WATSONX_URL, REDIS_URL, ALLOWED_ORIGIN
  • fly deploy — builds on Fly infra, zero local Docker needed
  • Verify: nomvox-api.fly.dev/api/ping
  • Cold start: auto_stop_machines = stop → 2–4s first-request wake
Vercel — Next.js
  • cd frontend && npx vercel --prod
  • Set NEXT_PUBLIC_API_URL=https://nomvox-api.fly.dev
  • Auto-deploy on every push to GitHub main
  • vercel.json: cache headers for /favicon.ico
  • app/icon.png: App Router favicon — registered as /icon.png route
  • Verify: nomvox.vercel.app
CORS Configuration
  • ALLOWED_ORIGIN env var on Fly.io
  • Set to https://nomvox.vercel.app in production
  • Set to http://localhost:3000 in development
  • Updated on every new Vercel deployment domain
  • Go middleware: Access-Control-Allow-Origin header check
Key Adaptations
  • Fly.io auto_stop = stop: zero cost at idle, 2–4s cold start
  • Upstash Redis: handles Fly.io machine wake/sleep connection teardown
  • IAM token cached in Redis (not in-memory) to survive machine restarts
  • Sequential watsonx calls with 600ms delay to avoid 429 rate limit
  • Next.js App Router: app/icon.png for favicon (not Pages Router approach)

12 Developer Feedback on IBM Bob

Honest, constructive observations from building NomVox end-to-end with IBM Bob as the primary development tool. Full detailed feedback in developer-bob-feedback.md.

✦ What Bob Did Well

  • Documentation friendly — README, SDLC plans, API tables: all publish-ready output, saved days of writing
  • Debugging autonomy — methodical diagnosis, traces execution paths across multiple files simultaneously, explains reasoning clearly
  • Teaching through doing — IBM watsonx IAM flow, Go goroutines, Next.js App Router hydration, archive/zip streaming — all genuinely understood post-build
  • Multi-stack fluency — held Go, TypeScript, Redis, Fly.io, and Vercel config in context simultaneously without losing context
  • Prompt engineering — 5-strategy naming system prompt, anti-name reasoning, brand persona card all produced noticeably better LLM output

✧ Friction Points

  • watsonx.ai setup confusing — IAM token flow, WML service association, Project vs Space ID, regional endpoint differences all underdocumented
  • IBM has no image generation model — serious gap for Creative Industries theme; Bob never resolved this, offered CSS fallback only after exhausting Imagen/SVG paths
  • HTTP 429 — architectural, not a bug — ~40 bobcoins diagnosing concurrent watsonx calls before root cause accepted
  • Favicon saga — ~40 bobcoins, 5+ approaches, final answer "it's correct, it will work on Vercel" — unsatisfying resolution
  • Outdated patterns — Next.js 13 patterns, Tailwind v3 syntax, old Fly.io CLI flags all required correction rounds
  • Unintended refactors — surgical requests for one change triggered rewrites of adjacent functioning code multiple times
  • Bobcoin efficiency — simple questions triggered long reasoning chains; recap overhead within sessions; large file token fragmentation
  • Training data ceiling — Puppeteer on Fly.io, Fly.io cold-start + Redis behavior: Bob's answers were generic when specificity was needed

Signal Sent. Brand Received.

Building NomVox was a genuinely rewarding experience. In roughly 15 days, one developer went from a blank canvas to a deployed, full-stack AI web application — spanning Go backend, Next.js frontend, Redis session layer, real availability checking across 6 platforms, a conversational AI loop, and a complete visual identity system.

The process taught real lessons that no tutorial covers: the actual mechanics of IBM watsonx IAM authentication, why training data quality is the ceiling of LLM output, why concurrent goroutines need careful timeout management, how Next.js App Router handles hydration differently from the Pages Router, and why CSS fallbacks are a design decision, not a consolation prize.

IBM Bob made the journey possible. It was a genuine creative and technical partner — not perfect, not always the most efficient, but consistently present, consistently knowledgeable, and genuinely educational. The limitations are documented honestly. The value was real.

The name NomVox itself came from an AI-assisted creative session exactly like the ones this tool enables. Nomen (name) + vox (voice). The cosmos names things. Now so do you.
